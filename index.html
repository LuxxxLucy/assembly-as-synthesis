TL;DR  Ordering blocks so the partial tower never topples is a program synthesis problem: search for a program (here, a placement order) that satisfies a specification (here, physics). Modern synthesis splits that search into a proposer that guesses and a verifier that checks; the proposer can be a SAT/SMT solver, a random sampler, an LLM, or a VLM. Same loop, different proposer. Five of them, on one ten-block task, side by side.

Two ways to do it

Program synthesis has a classical recipe. Encode the problem as logical constraints, hand them to an SMT solver — a theorem prover that decides satisfiability over theories like integers, arrays, and linear arithmetic — and read the plan off its model. SketchSolar-Lezama, Program Synthesis by Sketching (PhD thesis, Berkeley, 2008). The canonical modern synthesis system. did this for partial programs; FlashFillGulwani, Automating String Processing in Spreadsheets Using Input-Output Examples (POPL 2011). Shipped in Excel. did it for spreadsheet string transformations. The craft is in the encoding.

The classical route buys a strong guarantee: the solver either returns a plan or proves none exists. The price is paid in priors. A SAT/SMT solver does not know that bottom blocks go first, that arches need mutual thrust, that a heavy block on a narrow base is a bad idea. It knows only what the encoding says. Nothing accelerates the search beyond what the user wrote down by hand.

The second route runs on priors instead of proofs. A large language model is a statistical proposer that has read an enormous amount about how people solve physical problems — pyramids, scaffolding, keystones. Hand it the task and it often proposes a workable plan on the first try. When it doesn’t, the same verifier that rejected Z3’s plan can reject the LLM’s. The rejection message, in English, is exactly the shape of feedback the model was trained to consume.

Both routes run the same loop. It is called counterexample-guided inductive synthesis, or CEGISSolar-Lezama et al., Combinatorial Sketching for Finite Programs (ASPLOS 2006).: propose a candidate, verify it, and when verification fails, distill the failure into a reusable constraint before proposing again. The verifier does not just say no — it hands back a reason the next proposal must respect. That is the whole trick, and it is what makes a random sampler and a solver comparable on the same axis.

The task & the verifier

The task is to output the placement order for a target arrangement of blocks. Every prefix of the order has to stand up, and every next block has to reach its slot from above without clipping the ones already placed.

The verifier is the physics referee. For each step of a candidate plan it answers one question: can this block be placed right now, given what is already down? The verifierImplemented in src/counterplan/verifiers/. is a chain of three cheap physical checks, run cheapest-first so any failure short-circuits:

  • Kinematic — the block must descend into position without intersecting any block already placed (swept polygon test).
  • Stability — the partial structure must stand under gravity, friction, and compression-only contact (LP static equilibrium).
  • Landing — the target slot must actually catch the block; it should not fall past.

Each failed check returns a precedence constraint — a fact of the form “A must be placed before B”, with the physical reason attached. The stability check, for instance, solves a small LP and reports the minimal support set: the smallest set of still-missing neighbours whose presence would restore equilibrium. Each neighbour in that set becomes one precedence constraint. This is the CEGIS counterexample, made durableThe same move as CDCL clause learning in modern SAT solvers.: one failed round prunes an exponential slice of future orderings.

The proposer never touches physics. It needs only the current set of precedence constraints — in SMT, or the same constraints in English — and outputs a legal permutation. Whatever proposes that permutation plugs into the same loop.

1 · Random proposer — the baseline

The simplest proposer is Kahn’s topological sort with random tie-breaking — sample any permutation that respects the precedence constraints learned so far, hand it to the verifier, fold whatever new constraints come back into the next round’s pool. If those constraints ever form a cycle, the structure is inherently infeasible. An arch is the canonical case: no voussoir can go in first without a scaffold.

On pyramid_10, random CEGIS solves the problem in roughly seven rounds, learning about ten precedence constraints along the way. Against a 3.6 × 106 permutation space that is obviously a win; the question is how much further we can push things by changing only the proposer.

2 · SMT proposer — Z3

The plan is the model. We declare one integer variable per step — action[k], the block placed at step k, for k = 0..n−1 — and ask Z3 to assign them. Whatever assignment it returns is the plan. No decoder, no post-processing: the solver’s model is the artefact we wanted.

The baseline theory is trivial — two lines of SMT:

\[ \begin{aligned} &\mathsf{action}[k] \in \mathsf{BlockIds}, \quad k = 0, \dots, n-1 \\[2pt] &\mathsf{Distinct}\bigl(\mathsf{action}[0],\,\dots,\,\mathsf{action}[n-1]\bigr) \end{aligned} \]

Every learned precedence “A must precede B” becomes one clause, universally quantified over stepsQuantifier-free once unfolded over the fixed n. No triggers, no model-based quantifier instantiation — it is a plain propositional unfolding that Z3 handles directly.:

\[ \forall\, k.\ \mathsf{action}[k] = B \;\Longrightarrow\; \exists\, k' < k.\ \mathsf{action}[k'] = A \]

The loop is exactly the diagram above, instantiated for Z3. Z3 returns a model, we read the plan straight off the action[k] variables, the verifier runs, every fresh precedence becomes another clause, and we call check() again. If Z3 ever returns unsat, the structure is provably infeasible — not “we gave up”. That proof of impossibility is what the classical route buys you.

We don’t take this step here, but: one could bake the verifier itself into the theory. Stability is an LP; LP is a first-class SMT theory (QF_LRA). Kinematic clearance is a polygon-containment predicate; equality-saturation on the contact graph could give the solver an e-graphWillsey et al., egg: Fast and Extensible Equality Saturation (POPL 2021). of equivalent partial structures so deductions carry across branches. All of it is classical synthesis machinery, and all of it eventually runs out. Without priors, you are always exploring.

3 · LLM proposer — via a tool call

For the LLM route the protocol is agentic. The model sees one tool: submit_plan(plan: list[int]). Each turn it calls the tool with a full permutation of block ids; our code intercepts the call, runs the verifier, and returns the verifier’s feedback as the role: tool response. The model then gets another turn with that feedback appended to its context — no prose conversation history to reconstruct, no JSON parsing to chase, no hallucinated tool names.

The comparison

Same task (pyramid_10), same verifier, five proposers, one run each.A single run per proposer. The LLM calls are mildly stochastic; re-running may shift LLM/VLM lines by a round. The metric is best-so-far block similarity: the fraction of blocks the proposer has ever successfully placed before a verifier failure, tracked as the round counter advances. 1.0 means the proposer has found a fully feasible plan. It is the “best-so-far vs. trials” plot you see in black-box optimisation papers.

4 · VLM proposer — show, don’t describe

Every text prompt in the previous section encoded a choice: centroid vs. vertex list? World-frame or relative? Neighbours as an adjacency list or a graph drawing? Each choice is a guess about what the model needs.

The visual-language case cuts this. Render the target structure to a labelled PNG — the picture below — and hand it over with the same submit_plan tool. The first user message is the image; the blocks carry their permuted ids painted on top. Subsequent turns carry the same verifier-feedback tool responses the text LLM gets.

With kimi-k2p5Kimi-K2p5 supports chat, image input, and tool calls — the three things the loop needs. Fireworks serves it through the OpenAI schema. as the proposer, pyramid_10 one-shots: the first tool call is already a feasible plan (run_vlm.json). The VLM does not need to be told which block sits on the ground — it can see the ground.

Closing

The verifier is the load-bearing piece. Once a failure can be distilled into a durable, transferable constraint, CEGIS stops caring who does the proposing. A week of Z3 encoding gets you there. A paragraph of prose handed to a frontier model also gets you there. They are not competing paradigms — they are two settings of the proposer knob.

What I find more interesting is the same split one level up: use SMT to design the verifier, and an LLM or VLM to propose the plan. The formal side gets what benefits from proof; the statistical side gets what benefits from priors. Neither has to reach across.
